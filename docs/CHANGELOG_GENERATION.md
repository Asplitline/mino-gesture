# 更新日志（CHANGELOG）生成说明

应用内「设置 → 更新日志」**不手写 React 列表**：构建时从仓库根目录 **`CHANGELOG.md`** 读入原文，由 `src/lib/parseChangelogMarkdown.ts` 解析后渲染。

你在对话里**手动触发**时，把本文件（或 `CHANGELOG.md`）一并 @ 给助手，并用类似话术即可：

- 「按 `docs/CHANGELOG_GENERATION.md` 的约定，根据 `git log` 和当前代码重写 `CHANGELOG.md`。」
- 「只更新 `CHANGELOG.md` 里 `[x.y.z]` 这一节，与 `package.json` 的 version 对齐。」

助手应按下文「建议步骤」维护 **`CHANGELOG.md`**（必要时顺带核对 `package.json` 与 `src/appMeta.ts` 的版本号）。

---

## 文件与代码位置

| 用途 | 路径 |
|------|------|
| 唯一正文源 | `CHANGELOG.md` |
| 解析器 | `src/lib/parseChangelogMarkdown.ts` |
| 界面 | `src/pages/panel/components/settings-sheet/ChangelogSection.tsx` |

---

## `CHANGELOG.md` 书写约定（解析器依赖）

1. **版本节**（一级功能分组）：独占一行，格式严格为：

   ```markdown
   ## [主版本.次版本.修订] - YYYY-MM-DD
   ```

   日期可省略，但推荐保留，便于对照发布日。

2. **小节**（二级）：每个版本节下固定两类，**顺序为 `功能` → `修复`**（与常见发行说明一致），标题独占一行：

   ```markdown
   ### 功能

   ### 修复
   ```

   - **`### 功能`**：用户可见的新能力或行为改进，每条一行 `- `。
   - **`### 修复`**：缺陷修复；若无对应项，保留一节并写一条 `- 无。`，避免空节。

3. **条目**：小节标题下面，每条一行，**必须以 `- ` 开头**（无序列表）。解析器只认这种行；除上述两节外勿再增加其它 `###`，以免与「常规日志」约定不一致（确有需要时再扩展并同步解析说明）。

4. **可选强调**：正文里可使用 `**词**`，界面会去掉星号只保留文字。

5. **页脚引用链接**（如 `[0.1.0]: https://...`）可保留在文件末尾；解析前会自动剥掉，不影响版本内容。

6. **注释**：可在文件顶部用 HTML 注释写给人看的说明，解析时会整段删除。

---

## 助手生成时的建议步骤

1. 读 `package.json` 的 `version`，与 `CHANGELOG.md` 最新一节 `[x.y.z]` 对齐；若发新版，**新增一节** `## [新版本] - 日期`，旧节不要删。
2. 执行 `git log --oneline -80`（或更广范围），将变更分别归入 **`### 功能`** 与 **`### 修复`**。
3. 每个版本只维护 **`### 功能`** 与 **`### 修复`** 两节；`git log` 里 `feat` 类归纳进功能，`fix` 类归纳进修复。条目宜少而清楚，避免实现细节堆砌。
4. 保存后本地跑 `pnpm exec tsc --noEmit` 与设置页打开「更新日志」看一眼排版。

---

## 与 `package.json` 不一致时

若最新一节版本号与当前 `APP_VERSION`（来自 `package.json`）不一致，设置页会在更新日志顶部显示一行**版本不一致提示**，便于发现漏改。

---

## 实现备忘（维护解析器时）

解析器**不要用**带 `m` 标志的正则，以 `(?=…|$)` 截断到「下一版本或文末」：在 JavaScript 里多行模式下 `$` 表示**行尾**，正文会在第一行就被截断，导致 UI 几乎空白。当前实现改为按 `## [` 标题出现位置**切片**正文。
